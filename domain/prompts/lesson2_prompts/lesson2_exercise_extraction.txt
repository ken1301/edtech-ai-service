You are an expert educational content structurer and pedagogical designer. Your task is to parse a markdown problem file and populate the Exercise schema defined below, producing the 4-problem set for Lesson 2.

<lesson_continuity>

- Problem 1 (REINFORCEMENT) MUST be solvable using only the core knowledge and skills from Lesson 1 — do not require anything the student has not yet been taught.
- Later problems (CHALLENGE → EXPLORATION → EXTENSION) progressively raise difficulty and may stretch beyond Lesson 1, but each should still trace back to the same Concept.
- If <lesson1_summary> is empty, infer the foundation from the problems themselves.
</lesson_continuity>

<schema>
  <model name="Exercise">
    <field name="problem_list"     type="List[Problem]"  />
    <field name="subject"          type="Subject"         note="infer from content context" />
    <field name="topic"            type="Topic"           note="infer from content context" />
    <field name="concept"          type="Concept"         note="infer from content context" />
  </model>

  <model name="Problem">
    <field name="problem_id"               type="int"                 note="sequential integer starting at 1" />
    <field name="question"                 type="str"                 note="the full problem statement" />
    <field name="attachment_url"           type="Optional[str]"       note="null if not present" />
    <field name="approach_list"            type="List[Approach]"      note="extracted + newly generated approaches" />
    <field name="final_answer"             type="str"                 note="the canonical correct answer" />
    <field name="open_approach"            type="bool"                note="true if multiple valid approaches exist" />
    <field name="recommended_problem_role" type="ProblemRole"         />
    <field name="max_approach_trial"       type="int"                 note="equal to the total number of approaches in approach_list" />
  </model>

  <model name="Approach">
    <field name="summary"           type="str"                    note="one-sentence description of the approach" />
    <field name="bloom_level"       type="BloomLevel"             />
    <field name="concept_type_used" type="List[ConceptType]"      />
    <field name="pattern"           type="ExercisePattern"        />
    <field name="approach_answer"   type="str"                    note="the answer produced by following this specific approach" />
    <field name="strengths"         type="List[ApproachStrength]" />
    <field name="weaknesses"        type="List[ApproachWeakness]" />
    <field name="max_attempts"      type="int"                    note="estimated fair number of tries before giving a hint; typically 1–3" />
  </model>

  <model name="ExercisePattern">
    <field name="cognitive_operation" type="List[CognitiveOperation]" />
    <field name="representation"      type="List[Representation]"     />
    <field name="constraints"         type="List[Constraint]"         />
  </model>

  <enums>
    <enum name="BloomLevel"          values="remember | understand | apply | analyze | evaluate | create" />
    <enum name="CognitiveOperation"  values="recall | apply | analyze | transfer | construct" />
    <enum name="Representation"      values="symbolic | virtual | verbal | numeric | procedural" />
    <enum name="Constraint"          values="domain | tool | perspective | composition | none" />
    <enum name="ConceptType"         values="definition | formula | method | property | application | common_mistake | visualization" /> 
    <enum name="ApproachStrength"    values="optimal_complexity | easy_to_implement | generalizable | exact_result | handles_edge_cases" /> <-- can be none if not applicable -->
    <enum name="ApproachWeakness"    values="high_time_complexity | high_space_complexity | hard_to_implement | case_specific | approximation_only" /> <-- can be none if not applicable -->
    <enum name="ProblemRole"         values="reinforcement | challenge | exploration | extension" />
  </enums>
</schema>

<instructions>
  <step order="1" label="Parse">
    Read the markdown content inside <input> carefully.
    Identify every distinct problem block. Each block contains: a question, one or more existing approaches, and (if stated) a final answer.
  </step>

  <step order="2" label="Extract existing approaches">
    For each approach already present in the markdown, map it to the Approach model.
    Infer bloom_level, concept_type_used, pattern fields, strengths, and weaknesses
    from the approach's explanation and solution steps.
    Do NOT skip or merge approaches that are already present in the source.
  </step>

  <step order="3" label="Generate new approaches" critical="true">
    After extracting all existing approaches, reason about whether additional valid
    approaches exist that were NOT mentioned in the markdown.

    Criteria for generating a new approach:
    - It must reach the same final_answer via a meaningfully different method
      (different CognitiveOperation, Representation, or mathematical technique).
    - It must be pedagogically valuable (e.g., more generalizable, simpler, or
      illustrates a different concept type).
    - Do not generate trivially similar variations (e.g., same method with a sign change).

    For each new approach you generate:
    - Write a clear, step-by-step approach_answer.
    - Fill in all Approach fields just like extracted ones.
    - Add a note in the summary starting with "[Generated]" to distinguish it.
    - Append it to approach_list after the extracted approaches.

    If no additional valid approach exists, do not fabricate one.
  </step>

  <step order="4" label="Set problem-level fields">
    - Set open_approach to true if approach_list contains more than one approach.
    - Set max_approach_trial to len(approach_list).
    - Assign recommended_problem_role based on difficulty and pedagogical intent:
        * reinforcement  → familiar pattern, used to consolidate and review learned knowledge
        * challenge      → familiar pattern but with increased difficulty or added complexity
        * exploration    → novel or unfamiliar pattern designed for the student to discover a new pattern on their own
        * extension      → similar in spirit to exploration, offered as a follow-up to satisfy students who have completed an exploration problem
    - Infer final_answer from the markdown or from the approaches' shared conclusion.
  </step>

  <step order="5" label="Set exercise-level fields">
    Infer subject, topic, and concept from the problem content.
    These should be concise string labels (e.g., subject="Mathematics",
    topic="Quadratic Equations", concept="Completing the Square").
  </step>

</instructions>
